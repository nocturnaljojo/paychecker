# M0.5 UI Specification — Case-Card Upload Flow

> **Status:** Locked 2026-05-01 (Sprint M0.5-UI-SPEC). Binding contract for M0.5-BUILD-02 (UI redesign) and M0.5-BUILD-03 (vocabulary renderer + override flow).
>
> **Ratifies:** ADR-014 (document case paradigm).
> **Implements:** UI acceptance criteria of `docs/planning/M0.5-spec.md` (§"Acceptance criteria for M0.5-BUILD-01" + §"UI — `/upload` page redesign (M0.5)").
> **Source:** ChatGPT critique sessions 2026-05-01 (rounds 1–6).

---

## PART 1 — Design principles (locked, non-negotiable)

**PRIORITY RULE — MOMENTUM.** The worker must feel PROGRESS after every action. Even without calculation, the system must feel like: *"I am getting closer to understanding my pay."* Each interaction should make the worker feel they're moving forward, not stuck or testing the system.

**Three design principles:**

1. Clarity over completeness
2. Recognition over thinking
3. Momentum over perfection

**Critical rules:**

- NEVER block the worker (partial documents still useful)
- ALWAYS allow override (AI suggests, worker decides)
- NEVER force pre-classification (cognitive load too high)
- NEVER use dropdowns or text input (buttons only)
- NEVER use technical language
- NEVER promise what M0.5 doesn't deliver (no "Check my pay")

---

## PART 2 — Vocabulary (ratified, locked)

Worker-facing statuses (use ONLY these):

| Status | When to use |
|---|---|
| Ready | Enough pages to extract / check |
| Needs more pages | System thinks pages are missing |
| Can still use | Partial info usable, limited result |
| Not sure yet | Needs label or more pages |

**Worker action terms:**

- Confirm (or `✔ Yes`)
- Change (or `✏ Change`)
- Add more pages
- Continue

**FORBIDDEN words (must not appear in any worker-facing surface):**

- Failed, Error, Validation
- Classification, Confidence, Document type
- Match, Mismatch
- Needs review
- Any percentage or 0.0–1.0 score
- Step N of M (use count, not progress)
- Any technical jargon

---

## PART 3 — Continue button behavior (locked per ChatGPT round 6)

Tapping "Continue" must ALWAYS lead to visible progress.

**Acceptable destinations:**

- A new screen showing case count + progress state
- A clear next-step prompt ("Add another document")
- Visible change in case status (e.g., "ready" badge)

**NEVER acceptable:**

- Continue → same screen (no-op)
- Continue → blank waiting state
- Continue → "thank you" with no path forward

When worker taps Continue, design must answer: *"What did I just accomplish, and what's my next step?"*

---

## PART 4 — Progress feedback (locked per ChatGPT round 6)

Use count-based or state-based feedback only.

**Acceptable:**

- "1 document added"
- "2 documents ready"
- "✔ Contract confirmed"
- Simple icons / checkmarks

**NEVER acceptable:**

- Percentages (38%, 75%)
- Progress bars
- "Step 2 of 6"
- Spinning loaders without context

Apete doesn't think in %. He thinks in *"I've added something."* Use count + state, not percentage.

Layout (ASCII, mobile 375px width):

│ │                             │ │

│                                 │
│ ┌─────────────────────────────┐ │

│ │     📁                      │ │

- Slow connection: show button immediately, lazy-load icons
- First-time user: tip text orients them

### 5.2 — Upload in progress

When user taps Take Photo or Upload, then selects a file:

```
┌─────────────────────────────────┐
│ ← Back to your data             │
│                                 │
│ Check your work papers          │
│                                 │
│ ┌─────────────────────────────┐ │
│ │ [📄 thumbnail]              │ │
│ │                             │ │
│ │ Looking at your paper...    │ │
│ │                             │ │
│ │  ⏳                          │ │
│ │                             │ │
│ └─────────────────────────────┘ │
│                                 │
└─────────────────────────────────┘
```

**Stages (only show ONE at a time, transition smoothly):**

- "Uploading..." (during file upload, < 2s)
- "Looking at your paper..." (during AI classify, 2–10s)
- "Got it!" (transitions to confirmed card, 0.5s)

NO percentage. NO progress bar. Just thumbnail + state text + animated icon.

**If takes >15s, show:** *"This is taking a moment. Please wait."*

**If fails after 30s,** transition to 5.10 (low confidence card with manual override) — never show "Failed".

**Momentum checkpoint:** Worker feels: *"It's working on my thing right now."*

## PART 6 — Flow diagram (state transitions)

```
/dashboard
   │
   │ tap bucket card
   ▼

                                        │
CASE CARD CONFIDENT (5.3)                │
   │                                     │
   ├─ tap ✔ Yes ──→ CONFIRMED MOMENT (5.6) ──→ CASE LIST (5.5)
   ├─ tap ✏ Change ──→ OVERRIDE MODAL (5.7)
   │                                          │
   │                                          │ tap any category
   │                                          ▼
   │                                       CONFIRMED MOMENT (5.6)
   │                                          │
   ├─ tap + Add more pages ──→ ADD PAGES FLOW (5.8) ──→ CASE LIST (5.5)
   │
CASE LIST (5.5) (worker has multiple cases)
   │
   ├─ tap card ──→ CASE CARD (5.3 or 5.4)
   ├─ tap "+ Add another document" ──→ EMPTY STATE (5.1)
   │                                     (but with /upload?bucket=X
   │                                      preserved)
   └─ tap Continue (or nav from dashboard) ──→ NEXT STEP (5.9)
```

**Where state lives:**

- URL: `/upload?bucket={category}&case={case_id?}`
- Component state: `cases[]`, `currentlyUploading?`, `overrideOpen?`
- Server state: `document_cases` table (per ADR-014 schema in `M0.5-spec.md`)

## PART 8 — Mobile layout notes

**Tap targets:**

- All buttons: ≥48px tall (per Apete persona requirements)
- Minimum 8px gap between adjacent tap targets
- Card-style buttons preferred over text links (better thumb targets)

**Thumb zone:**

- Primary actions in bottom 1/3 of viewport
- Top 1/4 for navigation/title only
- Middle for content
- "Continue" / "Add another document" always at bottom

**Scroll behavior:**

- `/upload` page: scroll vertically, no horizontal scroll
- Multiple cases: scroll the case list, header sticky
- Modals: prevent body scroll while open

**Loading states:**

- Skeleton card while waiting for AI (5.2)
- Optimistic UI for confirmation (✔ Yes feels instant)
- Show 5.6 immediately, then sync to server

**Image handling:**

- Thumbnails: 80x80, gray-100 placeholder
- Lazy-load images on scroll
- Compress on upload to save bandwidth (rural WiFi)

**Network resilience:**

- Show cached cases immediately on load
- Optimistic UI for status changes
- Background sync; alert only if persistent failure

## PART 9 — Edge cases (handled)

| Scenario | UI behavior |
|---|---|
| Worker has 0 docs | Show empty state (5.1) |
| Worker has 1 doc, confirmed | Show confirmed card + Add another document |
| Worker has 1 doc, unconfirmed | Show full case card with buttons |
| Worker has 5+ docs | List view (5.5), unconfirmed first |
| Upload fails (network) | Show 5.10 with retry options, NEVER "Failed" |
| Upload succeeds, AI low conf | Show uncertain card (5.4) |
| Upload succeeds, AI 'other' | Show uncertain card (5.4) |
| Worker rejects all categories | Save as "Not sure yet", let them continue |
| Photo too blurry to read | Show 5.10 ("Hmm, this photo is hard to read") |
| Multiple uploads in flight | Queue them; show progress for each (NOT parallel) |
| Worker hits Continue with 0 confirmed | Suggest confirming or upload more first |
| Worker hits Continue with 1+ confirmed | Show 5.9 next-step screen |
| Bucket filter active (`?bucket=X`) | Filter list; show "Showing X documents" header |
| Existing case + new upload | Ask "Is this part of your X?" before classify |
| Slow connection (>30s) | Show "Still working..." after 15s; never give up |
| Worker re-uploads same image | Detect via `content_hash`; show "Already added" |

## PART 11 — Technical implementation notes

For tomorrow's M0.5-BUILD-02 sprint reference.

**Component architecture:**

- `/upload` page (route component)
- `<UploadZone />` — Take Photo / Upload buttons
- `<CaseCard />` — confirmed and unconfirmed variants
- `<CaseList />` — manages multiple case cards
- `<OverrideModal />` — slide-up modal for label change
- `<UploadProgress />` — pill states for in-flight uploads
