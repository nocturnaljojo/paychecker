# M0.5 Spec — Document Case Foundation

**Status:** Locked 2026-05-01 (Sprint UX-FLOW-AUDIT). Schema design committed; SQL applied at M0.5-BUILD-01 sprint start.
**Ratifies:** ADR-014 (`docs/architecture/decisions.md`).
**Supersedes:** Implicit per-document classification path of ADR-013 (path preserved as singleton-case fallback).
**Sub-phase of:** Phase 0 (`.claude/PLAN-PRJ-mvp-phases.md` §M0.5).

## Why this document exists

ADR-014 ratified the document-case paradigm but explicitly does NOT cover Migration 0012 SQL or the build-sprint task list. This spec is the binding contract for the M0.5 build sprint: the schema MUST match what is here, or this spec gets revised first.

## Scope (Option B — minimum-with-cases)

**IN scope:**
- Singleton case per upload (one upload → one case → one document, by default)
- `document_cases` table + `documents.case_id` FK
- `api/classify.ts` accepts array of `document_id`s; falls back to per-image when array length = 1
- `/upload` page redesign with inline case-card list
- Worker-facing case-card UI (`✔ Yes` / `✏ Change` / 5-bucket override / "+ Add more pages")
- Status vocabulary live (Ready / Needs more pages / Can still use / Not sure yet)

**OUT of scope (deferred to M1+):**
- 10-second proximity grouping heuristic (case-creation strategy stays singleton in M0.5)
- Multi-image bulk upload UI
- Visual + semantic continuity detection
- Multi-day session tracking
- Separate `/cases` index route
- Case archival + cross-case merging
- Calc engine (Phase 0 finish-line)

## Schema — Migration 0012 (locked)

```sql
-- M0.5 Migration 0012 — document cases as evidence-organization layer
-- Per ADR-014 (2026-05-01).

create extension if not exists "pgcrypto";

-- Worker-facing status vocabulary maps from this enum via the renderer.
-- Raw enum values never surface to workers.
create type document_case_completion_status as enum (
  'draft',       -- uploaded, not yet classified
  'suggested',   -- AI labelled, awaiting worker confirmation
  'confirmed',   -- worker accepted AI label OR overrode
  'partial',     -- confirmed but missing pages, can still use
  'complete'     -- all expected pages present, ready for calc
);

create table document_cases (
  case_id                   uuid primary key default gen_random_uuid(),
  worker_id                 uuid not null references workers(id) on delete cascade,
  doc_type                  text,                                  -- AI suggestion (post-classify)
  worker_confirmed_label    text,                                  -- override of AI suggestion
  completion_status         document_case_completion_status not null default 'draft',
  missing_items             jsonb,                                 -- e.g., {"last_page": true}
  confidence                numeric(3,2),                          -- latest AI classification confidence (operator-only)
  created_at                timestamptz not null default now(),
  updated_at                timestamptz not null default now()
);

-- documents.case_id FK — nullable so existing rows survive without backfill in M0.5.
-- M0.5-BUILD-01 may opt to backfill singleton cases for existing documents.
alter table documents
  add column case_id uuid references document_cases(case_id) on delete set null;

create index idx_document_cases_worker_status
  on document_cases (worker_id, completion_status);

create index idx_documents_case_id
  on documents (case_id);

-- RLS: same shape as documents — signed-in worker reads own; service role writes.
alter table document_cases enable row level security;

create policy document_cases_select_own
  on document_cases for select
  using (worker_id = current_worker_id());

create policy document_cases_update_own
  on document_cases for update
  using (worker_id = current_worker_id())
  with check (worker_id = current_worker_id());

create policy document_cases_delete_own
  on document_cases for delete
  using (worker_id = current_worker_id());

-- INSERT path is service-role-only (the upload pipeline creates cases server-side).
-- Workers never insert cases directly; they upload documents and the server creates the case.

-- updated_at trigger
create trigger document_cases_set_updated_at
  before update on document_cases
  for each row execute function set_updated_at();
```

**Notes:**
- `confidence` is captured for operator debugging only; never surfaces worker-facing (per ADR-013 + ADR-014 vocabulary lock).
- `missing_items` is an operator hint; the worker-facing rendering uses the status label, not the raw missing-items list.
- `worker_confirmed_label`, when non-null, takes precedence over `doc_type` in the worker view.
- `documents.batch_id` is preserved unchanged — orthogonal to `case_id` (batch = upload session; case = logical document).

## Endpoints — `api/classify.ts` shape change

**Existing input shape (per-document, ADR-013):**

```ts
POST /api/classify
{ "document_id": "uuid" }
```

**New input shape (per-case, ADR-014):**

```ts
POST /api/classify
{ "document_ids": ["uuid", "uuid", ...] }   // array of 1+
```

**Response (unchanged at output):**

```ts
{
  "case_id": "uuid",                       // case created or referenced
  "detected_type": "payslip|contract|super_statement|bank_export|shift|other",
  "confidence": 0.0-1.0,
  "page_breaks": [...],                    // populated when array length > 1 + mixed_content
  "mixed_content": true|false,
  "employer_guess": {"name": "...", "abn": "...", "confidence": 0.0-1.0}
}
```

**Server-side behaviour:**
- If `document_ids.length === 1`: create singleton case (1 doc), classify the single image (existing per-image semantics preserved).
- If `document_ids.length > 1`: create case, link all docs via `documents.case_id`, classify the multi-image set.
- All docs in the array MUST belong to the same `worker_id` (server-side enforcement; reject otherwise).
- Per `extraction-service-v01.md` versioning rules, this is a `classify-prompt-v02` bump.

## State machine (DB enum + renderer mapping)

**DB enum transitions:**

```
draft     → suggested   (AI classifies — server writes detected_type + confidence)
suggested → confirmed   (worker taps ✔ — worker_confirmed_label = doc_type)
suggested → confirmed   (worker taps ✏ + selects label — worker_confirmed_label = chosen)
confirmed → partial     (status: missing pages flagged)
partial   → complete    (more pages added, sufficient for extraction)
complete  → calc-ready  (extracted, in scope for engine — M0.5 stops here; calc is Phase 0 finish-line)
```

**Renderer mapping (worker-facing):**

| Internal enum | Worker-facing status |
|---|---|
| `draft` | Not sure yet |
| `suggested` | Not sure yet |
| `confirmed` | Ready (default) |
| `partial` | Needs more pages OR Can still use (worker preference toggle) |
| `complete` | Ready |

The Ready label appears for both `confirmed` (label-confirmed; M0.5 stops here) and `complete` (calc-ready; M0.5 cannot reach this state because no extraction in M0.5). M1+ extraction graduates `confirmed` → `complete`; the Ready label persists across both states.

## UI — `/upload` page redesign (M0.5)

Upload zone (top, full-width) + case-card list (below, scrollable). One case = one card. Order by `updated_at DESC` (newest cases at top).

**Case card sections (top to bottom):**

1. Document thumbnail strip (1+ thumbs depending on number of documents in case)
2. *"Looks like: {Contract / Payslip / Bank / Super}"* (or "We're not sure what this is" for low-confidence)
3. *"Status: {worker-facing label}"*
4. Action row: `✔ Yes` (full-width primary) and `✏ Change` (secondary)
5. Tertiary: *"+ Add more pages"* (links new upload to this case)

**Override flow (✏ Change):**

```
What is this?
[ Contract ]
[ Payslip ]
[ Bank ]
[ Super ]
[ Other ]
```

Big buttons. No dropdown. No text input.

**Bottom action bar (sticky), enabled when ≥ 1 case has `confirmed`/`partial`/`complete` status:**

```
[ Continue → Check my pay ]
```

In M0.5 this button routes to a placeholder ("Calc engine ships in Phase 0 finish-line"); the path is wired so M1+ can swap the destination.

## Acceptance criteria for M0.5-BUILD-01

A worker (Apete-shape, ESL, mobile, ≥48px tap targets) can:

1. Upload a single document
2. See a case card with an AI-suggested label and a worker-facing status
3. Tap `✔ Yes` to confirm the suggestion → case status moves to "Ready"
4. OR tap `✏ Change`, pick a different bucket → case status moves to "Ready" with overridden label
5. Tap "+ Add more pages" → upload a second document into the same case → see two thumbnails on the card
6. Refresh the page → case state persists
7. NEVER see the words: failed, validation, confidence, classification, error, needs review, document type, match, mismatch, or any percentage / 0.0–1.0 score
